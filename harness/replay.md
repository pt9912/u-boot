# Replay and Golden Sets

Diese Datei uebertraegt Replay-/Golden-Set-Denken auf die
`u-boot`-CLI-Generatoren. Ziel: Generator-Aenderungen sollen nicht nur
punktuell gruen sein, sondern gegen stabile Projektzustaende erneut
abspielbar bleiben.

## Scope

Replay gilt fuer alle Pfade, die User-Artefakte erzeugen oder
aktualisieren:

- `u-boot init`
- `u-boot init --template ...`
- `u-boot generate changelog`
- `u-boot generate readme`
- `u-boot generate env-example`
- `u-boot generate devcontainer`
- Add-on-Generatoren fuer `compose.yaml`, `.env.example` und extraFiles

## Golden Case

Ein Golden Case beschreibt einen reproduzierbaren Generatorlauf:

```text
Name:
Input project state:
Command:
Expected action:
Expected changed files:
Expected generated bytes or structural shape:
Expected idempotency result:
Expected safety behaviour:
Evidence test:
```

Nicht jeder Golden Case braucht externe `testdata`-Dateien. Kleine,
fokussierte Cases duerfen im Test selbst als In-Memory-Fixture stehen.
Externe Goldens sind Pflicht, wenn ein ganzes user-facing Artefakt
stabil bleiben soll oder wenn ein Diff im Test schwer lesbar wird.

## Replay Rules

- Jeder neue Generator oder jedes neue user-facing Artefakt bekommt
  mindestens einen Golden Case fuer Fresh-State und Double-Run-NoOp.
- Jeder managed-block-Generator bekommt einen Golden Case fuer stale
  block replacement und einen fuer user content preservation.
- Jeder destruktive oder konflikttraechtige Pfad bekommt einen Golden
  Case fuer "no partial write".
- JSON/YAML-artige Artefakte pruefen Struktur und, wo user-facing
  relevant, stabile Sortierung.
- Whole-file Golden Snapshots werden normalisiert nur dort eingesetzt,
  wo Strukturpruefung nicht reicht. Normalisierung darf keine
  semantischen Unterschiede verstecken.
- Golden Updates sind eine bewusste Produktentscheidung. Der Commit
  oder Slice muss sagen, warum sich der erwartete Output geaendert hat.

## Verification Evidence

Wenn ein Slice Generatorverhalten aendert, enthaelt die
Verification-Evidence nach [`harness/verification.md`](verification.md)
zusaetzlich:

```text
Replay / Golden:
- Affected generators:
- Golden cases added:
- Golden cases updated:
- Golden cases replayed:
- Intentional output changes:
```

## When to Add External testdata

Lege externe Fixtures unter dem relevanten Paket an, zum Beispiel
`internal/hexagon/application/testdata/golden/<case>/`, wenn:

- ein erwartetes Artefakt laenger als ein kurzer Inline-String ist,
- mehrere Dateien zusammen einen Zustand bilden,
- ein CLI-/Generator-Output fuer Nutzer sichtbar stabil bleiben muss,
- ein Review sonst nicht erkennen kann, ob die Aenderung absichtlich ist.

Halte externe Goldens klein. Ein Golden soll einen Vertrag pinnen, nicht
ein komplettes Demo-Projekt konservieren.

## Review Hooks

Reviewer sollen bei Generator-Diffs fragen:

- Gibt es einen Fresh-State Golden Case?
- Gibt es einen Double-Run-NoOp Golden Case?
- Gibt es einen User-Content-Preservation Case?
- Sind Sortierung und Format stabil?
- Wurde ein Golden geaendert, ohne dass Slice oder Commit die
  Output-Aenderung begruendet?
